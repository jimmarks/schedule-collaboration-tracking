# v3.0.25 - Critical Debugging Release

## Issue Summary

v3.0.24 showed that:
- ✅ REST routes ARE being registered
- ✅ Permission check runs and succeeds  
- ❌ **But `get_user_groups()` callback NEVER executes**

Permission granted at 16:09:34, but no `get_user_groups() called` log. This means WordPress isn't invoking the route callback despite successful permission check.

## v3.0.25 Enhancements

### 1. REST Dispatch Logging (cors.php)
Added `rest_pre_dispatch` filter to log ALL REST requests:
```
FTT REST DISPATCH: GET /ftt/v1/groups
FTT REST DISPATCH: Already handled: NO
```
This will show if WordPress is routing the request correctly.

### 2. Route Registry Verification (rest.php)
Added check at `rest_api_init` priority 999 to verify route is in WordPress's route table:
```
FTT REST: /ftt/v1/groups IS registered in WordPress REST API
FTT REST: Route details: [callback, permission_callback, methods, etc.]
```
Or if missing:
```
FTT REST: /ftt/v1/groups NOT FOUND in registered routes!
FTT REST: Found route: /ftt/v1/events
FTT REST: Found route: /ftt/v1/children
```

### 3. Complete Diagnostic Chain
Now we can trace the complete flow:
1. **Route Registration** → `register_routes() called`
2. **Route Verification** → `/ftt/v1/groups IS registered`
3. **Request Dispatch** → `REST DISPATCH: GET /ftt/v1/groups`
4. **Permission Check** → `check_groups_permission() called → Permission GRANTED`
5. **Callback Execution** → `get_user_groups() called` ← **Currently missing!**

## What This Will Reveal

### Scenario A: Route Not in Registry
```
FTT REST: /ftt/v1/groups NOT FOUND in registered routes!
```
**Cause:** Another plugin overriding or route registration failing  
**Solution:** Check for plugin conflicts, namespace collision

### Scenario B: Dispatch Shows Different Route
```
FTT REST DISPATCH: GET /ftt/v1/sync-children-to-groups
```
**Cause:** URL rewriting or routing issue  
**Solution:** Check URL formation, WordPress permalink settings

### Scenario C: Pre-dispatch Already Handled
```
FTT REST DISPATCH: Already handled: YES
FTT REST DISPATCH: Pre-dispatch result type: array
```
**Cause:** Another filter returning early, preventing our callback  
**Solution:** Check filter priorities, look for intercepting code

### Scenario D: Route Registered, Dispatch Works, But Callback Silent
```
✅ /ftt/v1/groups IS registered
✅ REST DISPATCH: GET /ftt/v1/groups → Already handled: NO
✅ Permission GRANTED
❌ get_user_groups() never called
```
**Cause:** Callback configuration issue or PHP error before first log  
**Solution:** Check callback array format, look for early PHP fatal

### Critical Questions v3.0.25 Will Answer

1. **Is `/ftt/v1/groups` in WordPress's route registry?**
   - If NO: Route registration failing completely
   - If YES: Route exists but callback not executing

2. **What route does WordPress see?**
   - Look for `REST DISPATCH:` line to see actual route
   - Compare to expected: `GET /ftt/v1/groups`

3. **Is something handling the request early?**
   - `Already handled: YES` means `rest_pre_dispatch` filter returned non-null
   - Some other code intercepting before our callback runs

4. **What does the route details show?**
   - Should show: `'callback' => ['FTT_REST', 'get_user_groups']`
   - If different: Callback configuration corrupted
